部署方案已分析完成，建议采用“最小改造、保留现有 API 逻辑”的方式把项目变成可在普通 Linux 服务器运行的常驻 Node 服务。

影响范围与副作用分析：
1. 受影响文件
   - [`package.json`](package.json:1)：新增服务器启动脚本，可能补充服务端依赖
   - 新增 `server.js`：作为 VPS 常驻入口，负责静态资源与 `/api/*` 路由分发
   - 可能新增一个轻量 HTTP 适配层文件，例如 `lib/http.js` 或 `lib/node-server.js`
   - [`DEPLOY_SERVER.md`](DEPLOY_SERVER.md:1)：改成真实可执行的 VPS 部署文档
2. 潜在副作用
   - 现有 [`api/tasks.js`](api/tasks.js:1)、[`api/winners.js`](api/winners.js:1)、[`api/auth/*.js`](api/auth/login.js:1) 目前假设 `req/res` 具备 Vercel 风格的 [`res.status(...).json(...)`](api/tasks.js:10) 能力，普通 Node 原生 [`http`](https://nodejs.org/api/http.html) 需要做兼容包装
   - 前端构建后需要由服务端托管 `dist/` 静态文件，否则服务器上只有 API 没有页面
   - Cookie、JSON body、查询参数在 Node 原生环境下都需要自行处理

方案比较：
1. 方案 A：引入 Express
   - 优点：实现快，路由/静态托管/body 解析现成
   - 缺点：新增依赖；对这个小项目略重
2. 方案 B：用 Node 原生 [`http`](https://nodejs.org/api/http.html) + 轻量适配层
   - 优点：依赖最少，部署更稳，适合 VPS
   - 缺点：需要自己写一点请求/响应封装
3. 结论
   - 选方案 B。当前项目 API 数量少，响应格式简单，做一个轻量适配层最合适，能最大化复用现有 [`api/`](api:1) 目录逻辑。

建议实施步骤：
1. 新增 `server.js`
   - 使用 Node 原生 HTTP 服务监听端口
   - 托管 `dist/` 下静态文件
   - 将 `/api/tasks`、`/api/winners`、`/api/auth/login`、`/api/auth/logout`、`/api/auth/session` 路由映射到现有 handler
   - 对 Node 原生 `req/res` 做 Vercel 风格兼容：`req.method`、`req.body`、`req.query`、`res.status()`、`res.json()`
2. 更新 [`package.json`](package.json:1)
   - 新增 `build` 后服务器运行脚本，如 `start` / `start:server`
   - 保留当前前端与 SQLite 导入脚本
3. 更新 [`DEPLOY_SERVER.md`](DEPLOY_SERVER.md:1)
   - 改成完整流程：修复 PATH、安装依赖、构建前端、导入 SQLite、用 PM2 启动 `server.js`、配置 Nginx
4. 可选验证点
   - `GET /api/auth/session`
   - `GET /api/tasks`
   - 前端首页 `/`

预计文件变更：
- 修改 [`package.json`](package.json:1)
- 新增 `server.js`
- 修改 [`DEPLOY_SERVER.md`](DEPLOY_SERVER.md:1)

这个方案不会改动核心业务存储逻辑 [`lib/sqlite.js`](lib/sqlite.js:1)，也不会重写现有 API，只是在外层补一个服务器入口与适配层，风险最低。
